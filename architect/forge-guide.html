Three Entry Points

Choose based on task complexity. Each can be used standalone or the orchestrator invokes the others.

/forge [goal]
Full orchestrator pipeline. Parses goal into milestones, runs research → build → review cycles, maintains state files for compaction survival.
Use when: Multi-step project, 2+ milestones, needs coordination between research and building
/forge-research [topic]
Autonomous research campaign. Multi-agent breadth → depth with hypothesis tracking, adversarial challenge, and source verification.
Use when: Deep technical research, comparing approaches, evaluating libraries — not building yet
/forge-builder [task]
Autonomous build with self-review, self-improvement, and quality gates. Classifies input as directive (build) or inquiry (analyze only).
Use when: Single feature or fix with clear scope. No research phase needed.

R  Research Phase

Invokes forge-research for unknowns in the current milestone. Research runs to completion before building starts. Skipped if the milestone has no unknowns.

Spawns: research-worker agents (web + codebase)

Output: architect/research/{topic}.md

P  Planning Phase (complexity-adaptive)
ComplexityApproach
Simple (single file, clear spec)Skip detailed planning, inline notes
StandardConcise approach with key decisions
Complex (multi-file, unknowns)Detailed plan with ≥2 approaches + trade-offs

Spawns: adversarial-reviewer to critique the plan (max 3 iterations until no CRITICAL/HIGH)

B  Build Phase

Invokes forge-builder with the refined plan. Builder executes with self-review and self-improvement loops. Waits for completion signal.

Spawns: build-worker in worktree isolation (Claude Code) or with scoped ownership (Codex)

State: TODO.md updated per phase, git checkpoint on gate pass

V  Review Phase (fresh context, parallel)

Critical: All reviewers run in FRESH context. They get only: milestone spec + git diff + test results. No builder conversation history. This avoids confirmation bias (40-60% quality improvement per ASDLC research).

BatchAgentsFocus
Batch 1 (up to 3 parallel)adversarial-reviewer + performance-auditorBugs, security, logic, performance, test coverage
Batch 2 (after batch 1)Documentation + Brainstorm agentsDocs for changes, improvement ideas

Output: architect/review-findings/{milestone}.md

2  Second-Opinion Phase (optional)

Try other CLIs if available (Codex, Gemini). Send git diff + intent. Common case: all unavailable → skip entirely, note in summary.

I  Improvement Phase

Consolidates all review findings. Classifies each:

TypeAction
Bug fixApply immediately
Improvement (existing)Apply
New featureWrite to SUGGESTIONS.md, do NOT implement

Re-runs quality gates after improvements applied.

G  Quality Gate

ALL must pass: tests green, critical findings addressed, no regressions, diminishing returns check.

OutcomeAction
PassProceed to COMPOUND
FailLoop back to BUILD (max 2 retries per milestone)
Fail after 2 retriesEscalate to user with details
C  Compound Step

After gate passes, codify learnings for future milestones:

  1. Update CLAUDE.md with conventions discovered this milestone
  2. Consolidate SUGGESTIONS.md (deduplicate, sort by confidence)
  3. Check SUGGESTIONS.md for human feedback (approved → new milestones)
  4. Update TODO.md with deferred items
  5. Update FORGE-MEMORY.md with learnings
  6. Update FORGE-STATUS.md + FORGE-HANDOFF.md
  7. Check HUMAN-INPUT.md + MISSION-CONTROL.md for directives
  8. Git commit milestone output
✓  Finalization

Runs after all milestones complete: full test suite, update CLAUDE.md, consolidate SUGGESTIONS.md, generate summary, cleanup temporary artifacts (agent-contexts/, review-findings/, DEFERRED-CHANGES.md).

/forge-research — Standalone Research

9-step research campaign with hypothesis tracking and adversarial challenge.

1. Parse research directive
Extract topic, questions, scope, constraints, output path
2. Platform detection
Read forge-claude-teams or forge-codex-multiagent
3. Formulate hypotheses + Questions to Resolve
Termination criteria: all questions must be checked off
iterates until diminishing returns or 3 depth cycles
4. Phase 1: Breadth agents
Web research (study skill) + codebase exploration
Parallel for independent queries, sequential for dependent
5. Synthesize findings
Findings table with confidence scores, update hypothesis status
6. Adversarial challenge
Self-critique (always) + adversarial-reviewer (if available)
Check: confirmation bias, source quality, gaps, contradictions
7. Phase 2: Depth agents
Targeted queries for gaps, cross-reference disputed claims
8. Final synthesis
Confidence scoring (high/medium/low), hypothesis resolution
9. Write output
architect/research/{topic-slug}.md

State Files & Human Steering

Persistent files that survive compaction, enable resumption, and allow human steering during autonomous runs.

Agent State Files

FORGE-STATUS.md

agent → disk

Current milestone, phase, active agents, completed/pending steps, blockers. Updated at every phase transition.

FORGE-HANDOFF.md

agent → disk

Compaction survival checkpoint. "What was I doing RIGHT NOW?" Updated every 5 steps + at milestones. ~600 tokens.

FORGE-MEMORY.md

agent → disk

Cross-milestone learnings. Minimum-signal gate: "Will a future agent act better knowing this?"

TODO.md

agent → disk

Task checklist with completion status. Updated after every significant step.

Human Steering Channels

SUGGESTIONS.md

bidirectional

Agent writes suggestions with confidence scores. Human writes: approved / rejected / needs-revision. Checked every phase transition.

HUMAN-INPUT.md

human → agent

New work items, brainstorming ideas. Agent processes and moves to "Processed" section.

MISSION-CONTROL.md

human → agent

Steering directives: PRIORITY, STOP, NOTE, FOCUS, EXPAND. The "steering wheel" for long-running sessions.

Compaction Survival

How the pipeline survives auto-compaction
  1. CLAUDE.md is the anchor — survives compaction verbatim (re-read from disk)
  2. ## Forge Pipeline State section instructs: re-read FORGE-HANDOFF.md, FORGE-STATUS.md, TODO.md, FORGE-MEMORY.md
  3. Agent reads state files, recovers full session context
  4. Resumes from the last completed phase transition
  5. Increments compaction_count in FORGE-HANDOFF.md
  6. At count 4-5: delegates more aggressively to sub-agents. At 5+: maximum delegation.

Custom Agent Roles

Four specialized agents with strict role boundaries. Read-only agents can't write; build workers can't exceed scope.

forge-adversarial-reviewer

read-only

Critical reviewer for code, plans, and specs. Finds flaws, gaps, ambiguities, and security issues.

Confidence gate: ≥80% with file:line evidence
Severity: S3C3=CRITICAL, S2C2=WARNING, S1C1=INFO

forge-build-worker

full-access

Implementation worker. Builds code within declared file scope, self-verifies against spec, runs tests after every change.

Scope: MUST NOT modify files outside ownership
Step limit: 60 tool calls max

forge-research-worker

read-only

Web research and codebase exploration. Source verification, hypothesis tracking. NO write access (Rule of Two).

Every claim must cite a source (URL or file:line)
Per-claim confidence rating required

forge-performance-auditor

read-only

Metric-driven performance auditing. Runs benchmarks, profiles code, reports with reproducible measurements.

Every finding needs exact reproduction command
Baseline + measured + threshold per metric
